## Critical Functional Findings

### F-01 — Mobile validation scroll ignores the fixed app bar

- Severity: P0
- Symptom: pressing the bottom action with an incomplete configuration moves the page by roughly the app bar height and leaves the active heading underneath the fixed app bar. The user sees a dark/partial state and cannot tell what is missing.
- Evidence: at `390x844`, after `Худі`, the bottom action shows `До зон`; without fit/fabric it keeps `config` active, sets the warning, and `scrollY` becomes `81` while the active step top is `-1px`.
- Root cause: `showStepProblem()` calls `field.scrollIntoView({block: "center"})` without accounting for the app bar and bottom bar. `setActiveStep()` and `navigateToStep()` also use raw `scrollIntoView()`.
- Fix direction: one `scrollToStudioTarget(element, {focus, reason})` helper using `window.scrollTo` and measured app bar/safe-area offsets; no raw `scrollIntoView` in the configurator transition paths.

### F-02 — Final actions are not the real submit controls on mobile

- Severity: P0
- Symptom: the user reaches the last section, sees grey disabled-looking `Додати в кошик` and `Надіслати менеджеру`, and the bottom action is the only obvious control. Clicking the visible final buttons can appear to do nothing when the policy blocks them.
- Root cause: `updateFinalActionsAvailability()` only toggles `disabled` and generic hints. The final panel has no per-field completion checklist and no `aria-describedby`/focus target explaining the missing requirement. The mobile bar owns the submit path while the visible final actions do not provide the same error reveal behavior.
- Fix direction: make both final buttons call the same guarded submit handlers immediately; render a live completion checklist with one row per missing requirement; focus/reveal the first missing step and keep the action visually actionable with a concrete reason.

### F-03 — Manager action does not carry a prepared configuration summary

- Severity: P1
- Symptom: the manager icon is a generic chat bubble and Telegram opens without a compact selection summary.
- Root cause: the dialog link uses the static manager URL; `openManagerDialog()` persists the draft but does not generate a prefilled, URL-safe summary message.
- Fix direction: use a help/person icon, build a localized summary from selected mode/product/fit/fabric/color/zones/format/quantity, preserve the draft, and append it to the Telegram deep link without exposing files or unsafe raw input.

## Visual Findings

### V-01 — Fit cards are oversized on mobile and stack vertically

- Severity: P1
- Symptom: classic and oversize hoodie cards show huge dashboard-style PNGs and consume almost the entire first screen. On mobile, two choices are vertical instead of left/right.
- Root cause: the late mobile CSS forces `.cp-fit-grid` to one column and the generated `.cp-fit-card-figure` has no compact dimensions; the card images dominate the card.
- Fix direction: two-column fit grid from `320px` upward, fixed thumbnail box, compact text hierarchy, selected ring/check and clear `aria-pressed` state. Keep the larger image available in the preview overlay instead of inside every selector.

### V-02 — Fabric choices use unconstrained flex wrapping

- Severity: P1
- Symptom: `Термо` drops to a visually detached lower row and the selected fabric is not centered with its peers.
- Root cause: `.cp-fabric-row` is a wrapping flex row while generated fabric chips have no shared grid track or min/max content policy.
- Fix direction: CSS grid with equal tracks (`repeat(2, minmax(0, 1fr))` for two options, `repeat(3, minmax(0, 1fr))` for thermo palettes where space allows); lock heights, clamp secondary copy, and use a dedicated info icon.

### V-03 — Fabric info trigger is an invalid nested interactive control

- Severity: P1
- Symptom: premium fabric shows a literal `?` inside a button; the click target is ambiguous and the question mark looks unfinished.
- Root cause: `<span role="button">` is nested inside the fabric `<button>`, with only a click handler and no keyboard behavior.
- Fix direction: use an icon-only `i`/info control with `tabindex=0`, keyboard activation and an accessible label; keep it visually aligned in the card header and stop propagation safely.

### V-04 — T-shirt palette is wrong for the requested product matrix

- Severity: P1
- Symptom: regular/oversize T-shirts expose grey/graphite and do not expose coyote. Thermo correctly needs its own green/pink palette.
- Root cause: `PRODUCT_MATRIX["tshirt"].colors` contains black/white/graphite; `renderColorChips()` only switches palettes when the selected fabric has `colors`.
- Fix direction: replace graphite with coyote (`#8B6B45`) in the T-shirt base palette; leave thermo colors as the fabric override. Add a regression contract for exact values.

### V-05 — Hoodie fit palette is not separated

- Severity: P1
- Symptom: oversize hoodie exposes the same broad palette as regular, while the requested interim matrix is black/pink for oversize. Premium appears as a mysterious included item with a question mark.
- Root cause: hoodie has one global `colors` list and no fit-specific palette. Oversize premium is auto-selected but the UI still renders it as if it were a normal choice.
- Fix direction: add `fit_colors` (`regular`: current broad palette, `oversize`: black/pink), resolve palette by fit, and render auto-included premium as a locked `Входить у базу` material badge instead of a selectable mystery control.

### V-06 — Fleece controls are custom-rendered without a stable selection model

- Severity: P1
- Symptom: `З флісом` / `Без флісу` look detached and may appear non-functional; the active state is not communicated as a binary setting.
- Root cause: `renderAddons()` creates a separate `.cp-fleece-toggle` inside the addon grid, while state is stored as mutually exclusive addon values and defaults to neither. The CSS has no coherent visual contract for the generated classes.
- Fix direction: render a two-option segmented control with explicit `aria-pressed`, default to the configured base choice, and keep the state in the existing `add_ons` payload.

### V-07 — Zones and front/back format selectors lack a shared visual language

- Severity: P1
- Symptom: front/back zones, A-size choices and sleeve options look like unrelated small rectangles; users cannot quickly understand what is selected or what the size means.
- Root cause: generated controls rely on generic card rules, while their SVGs/text have no fixed icon frame or dimensional hierarchy.
- Fix direction: add dedicated visual zone tiles with garment placement icons, a fixed format badge (`A4 · 21 × 29,7 см`), and selected/check/disabled states that work on light/dark colors.

### V-08 — Hero sample print frames are not consistently calibrated on narrow screens

- Severity: P1
- Symptom: hero print-zone rectangles and labels look uneven or detached on phones.
- Root cause: mobile uses hard-coded `left/top/width/height` overrides independent of the source image aspect ratio.
- Fix direction: anchor frames to the stage aspect-ratio container with percentages and a single calibration map; hide only nonessential labels below `380px`, never the geometry itself.

## Navigation And Layout Findings

### N-01 — Progress ownership changes during render

- Severity: P0
- Symptom: the eye/progress app bar appears to disappear or shift on some stages; the progress rail can be moved into an active section.
- Root cause: `syncProgressShellPlacement()` physically re-inserts the progress shell into the active step on mobile. Combined with hidden step wrappers and smooth scrolling, this creates stacking and ownership changes.
- Fix direction: keep one app bar portal under `document.body` for mobile and one desktop progress rail in a stable shell; do not move the same node between sections during state refresh.

### N-02 — SEO content can become part of the apparent app journey

- Severity: P1
- Symptom: after advancing or overscrolling, the user can land in the long SEO text and lose the sense of the constructor.
- Root cause: previous markup kept SEO within the same shell boundary and transitions used generic scroll behavior.
- Fix direction: preserve the existing outside-shell boundary and add a studio sentinel/scroll clamp so app navigation never targets the SEO stack; keep SEO available only after leaving the constructor.

### N-03 — Bottom action bar can outlive the meaningful studio content

- Severity: P1
- Symptom: continued downward scrolling reaches empty space or unrelated sections while the app UI no longer provides useful context.
- Root cause: the body remains freely scrollable after the active studio stage ends, while the app bar/bottom bar state is not tied to a studio boundary.
- Fix direction: add an explicit studio boundary observer; hide mobile bar and release app shell when the user leaves the configurator, and ensure the active step has enough bottom padding but no artificial blank tail.

### N-04 — App bar overlaps active content after validation errors

- Severity: P1
- Symptom: headings and first controls can sit underneath the app bar after a failed next action.
- Root cause: no `scroll-margin-top`/offset contract for focused fields and active steps.
- Fix direction: central scroll helper plus CSS `scroll-margin-top` on all active steps and interactive groups.

## Completion And Accessibility Findings

### A-01 — Incomplete final step is visually grey instead of diagnostic

- Severity: P1
- Root cause: final buttons only use `disabled`/grey styles; there is no checklist of missing format, file, quantity, sizes, contact or Telegram verification requirements.
- Fix direction: live checklist, explicit statuses (`Готово`, `Потрібно заповнити`), direct edit links and focus behavior.

### A-02 — Generated icon controls need keyboard parity

- Severity: P1
- Root cause: fabric info and some generated cards are `div/span` interactions without consistent keyboard handlers and labels.
- Fix direction: all controls must be real buttons or have complete keyboard semantics; icon-only controls get `aria-label` and visible focus.

### A-03 — iOS zoom is triggered by small form controls

- Severity: P1
- Symptom: tapping brief/contact fields on iPhone changes zoom and leaves the layout looking shifted.
- Root cause: generated inputs/textareas must be checked for a computed font size below `16px` on iOS.
- Fix direction: enforce `font-size: 16px` on mobile form controls and preserve visual scale via line-height/padding rather than browser zoom.
