## 1. Что уже исправлено и задеплоено (12.06.2026, ветка `fix/analytics-event-loss` → main → prod)

1. **view_item терялся почти всегда.** `product-detail.js → trackViewContent()` делал ранний `return` при отсутствии `window.trackEvent` (а он появляется только после загрузки analytics-loader.js, т.е. после первого взаимодействия). Событие не доходило даже до dataLayer. Теперь dataLayer-push безусловный.
2. **Ранний стаб `window.trackEvent` с очередью** (base.html + `setupGlobalEventBridge` в analytics-loader.js): все Meta/TikTok-вызовы до инициализации лоадера буферизуются в `window._trackEventQueue` и отправляются после загрузки. Раньше — молча терялись из-за guard'ов `if (window.trackEvent)`.
3. **begin_checkout для заказов через форму доставки** (наложка/COD): срабатывает один раз на `focusin` формы `#guest-form`/`#deliveryForm` (main.js → `attachBeginCheckoutOnFormStart()`), дедупликация с Monobank-кнопками через `window.__twcBeginCheckoutSent`.

**Задача мониторинга (для исполнителя):** через 7 дней после 12.06 сверить в GA4: view_item должен стать ≥ числа просмотров карточек (~300+/мес), begin_checkout ≥ 30–40/мес. Если нет — искать остаточные потери (Realtime + DebugView).

## 2. СРОЧНО (1–2 недели)

### 2.1. Двойной page_view: GTM-тег + ручной gtag (риск задвоения всей статистики)
- **Проблема:** page_view может считаться дважды → искажённые сессии/показатель отказов → мусорные данные для рекламных алгоритмов.
- **Причина (доказано конфигом):** в GTM live-версии тег `[38] GA4_Configuration_Tag` (googtag, G-109EFTWM05, **без** `send_page_view:false`, триггер Initialization) И параллельно analytics-loader.js:905 делает `gtag('config', GA_ID, {send_page_view:false})` + вручную `gtag('event','page_view')` (строка 908). Оба пишут в один dataLayer/один measurement ID.
- **Решение:** проверить в GA4 DebugView (включить debug_mode, открыть любую страницу, посчитать page_view). Если двоится — в GTM-теге 38 добавить параметр `send_page_view = false` (правильнее всего: один источник истины — оставить ручной page_view лоадера, т.к. он несёт корректный timing). Опубликовать новую версию контейнера.
- **Бизнес-эффект:** чистые данные сессий/конверсии — фундамент для Meta/Google Ads. **Сложность:** низкая. **Приоритет: P0.**

### 2.2. Clarity загружается дважды
- **Проблема:** GTM-тег `[58] Microsoft Clarity - Official` (projectId t7u94cvpqc) И analytics-loader.js `loadClarity()` грузят один и тот же проект. Двойная инициализация = лишний JS на мобильных + потенциальные конфликты сессий.
- **Решение:** удалить тег 58 из GTM (лоадер управляет загрузкой умнее: не грузит на low-end девайсах и на home до интеракции). Либо наоборот — оставить GTM-вариант и вырезать loadClarity из лоадера, но тогда теряется device-class логика. Рекомендация: **удалить из GTM**.
- **Эффект:** −~30KB JS на мобильных, консистентные записи. **Сложность:** низкая. **P0.**

### 2.3. Мусор в GTM: мёртвые теги, дубль-триггеры, UA-зомби на сервере
- **Проблема/факты:**
  - 5 paused-тегов FB_CONVERSIONS_API (18–22) + 8 переменных к ним — наследие старого CAPI-сетапа через GTM; рабочий CAPI давно в Django (`orders/facebook_conversions_service.py`). Paused-тег 18 ссылается на мёртвый transport_url `server-side-tagging-ljdnm3qrca-uc.a.run.app`.
  - Дубль-триггеры purchase: `[37] purchase` и `[39] Purchase` (оба `_event == purchase`); 37 не используется.
  - Server-контейнер `GTM-KQWHQGRL` содержит тег **Universal Analytics (sgtmua)** — UA мёртв с июля 2023, тег шлёт данные в никуда. Второй server-контейнер `GTM-P6V79HW6` пуст (1 клиент, 0 тегов).
- **Решение:** удалить paused FB-теги + их переменные + триггеры 3/4; удалить триггер 37; удалить/архивировать оба server-контейнера (server-side tagging сейчас не используется — CAPI работает напрямую из Django, это нормально). Опубликовать версию с понятным именем «Cleanup 2026-06».
- **Эффект:** прозрачный контейнер → меньше ошибок при будущих правках под Meta/TikTok Ads. **Сложность:** низкая. **P0.**

### 2.4. Внутренний трафик загрязняет GA4 (Direct 50%, /admin-panel/ 133 просмотра в GA)
- **Проблема:** ~половина «трафика» — собственные заходы владельца/разработки; /admin-panel/ виден в отчётах; desktop > mobile аномалия. Все CR/bounce-метрики занижены/искажены.
- **Решение (3 слоя):**
  1. GA4 Admin → Data Streams → Configure tag settings → **Define internal traffic** (IP офиса/дома) + Data Settings → Data Filters → активировать фильтр internal.
  2. В base.html не инициализировать аналитику, если `request.user.is_staff` (передавать флаг в data-атрибут `<html>` и выходить из лоадера) — это уберёт админ-сессии независимо от IP.
  3. Clarity: Settings → IP blocking те же IP.
- **Эффект:** реальные CR/bounce → правильные решения по деньгам. **Сложность:** низкая. **P0.**

## 3. СРЕДНЕСРОЧНО (1–2 месяца)

### 3.1. Custom Print 2.0 — Mobile-First конструктор (Блок 4 мастер-задачи)
- **Проблема:** конструктор десктоп-ориентирован; SVG-модель тяжела в поддержке и фрустрирует на тач; цена не всегда видна; шаги неочевидны. Данные: dead/rage clicks (п. 2.6), скролл 52%, конверсия страницы в лид требует замера по шагам.
- **Решение (архитектура):**
  1. **Step Wizard:** 1) формат (себе/команде) → 2) изделие+цвет → 3) принт (зоны: груди/спина/рукав) → 4) размер/кол-во → 5) контакт. Каждый шаг = отдельный экран на мобильном, прогресс-бар сверху.
  2. **Sticky bottom bar:** всегда видимые цена + кнопка «Далі». Цена пересчитывается на каждый input.
  3. **Замена SVG-модели:** сравнение вариантов: (а) PNG-рендеры по 6–8 ракурсов на изделие/цвет — дёшево, быстро, фотореалистично, рекомендую как этап 1; (б) canvas-композиция print-на-фото (drag/scale принта) — этап 2; (в) Three.js 3D — дорого, тяжело на low-end, отложить. Критерии: скорость загрузки < 1s, реалистичность, стоимость поддержки при добавлении изделий.
  4. **Комментарий к каждой зоне печати** (опционально, textarea per print zone: front/back/sleeve).
  5. **Трекинг по шагам:** dataLayer события `cp_step_1..5`, `cp_price_seen`, `cp_submit` → GA4 funnel exploration. Без этого нельзя оптимизировать.
  6. **Telegram-уведомления переделать:** структурированный шаблон (изделие, цвет, размер, зоны печати + файлы, формат, цена, контакт, UTM источника лида), Markdown-разметка, фото-превью конфигурации.
- **Эффект:** главный лидоген сайта; +30–50% к лидам при том же трафике — реалистичная цель. **Сложность:** высокая. **P1.**

### 3.6. Снижение трения в покупке (cart = checkout)
- **Факт:** корзина и checkout — одна страница (это хорошо, шагов мало). Но: 82 ATC → 4..N begin_checkout → 1 purchase. После фиксов событий замерить честную воронку и тогда решать. Кандидаты на улучшение: (1) автокомплит Новой Почты (города/отделения — есть ли уже?); (2) сохранение корзины гостя (localStorage уже есть для полей — проверить для items); (3) экспресс-оплата Apple/Google Pay прямо с карточки товара (mono checkout попробовать на PDP); (4) бейдж «оплата при отриманні» на карточке — для недоверчивых.
- **P2, после недели чистых данных.**

## 4. ДОЛГОСРОЧНО (2+ месяца)

### 4.1. B2B-страница с интерактивным выбором сегмента (Блок 6)
- Лендинг `/b2b/`: выбор «Я — дизайнер / дропшиппер / физмагазин / интернет-магазин / Prom / другое» → персональный оффер (свои выгоды, прайс-логика, CTA). Лид-форма с сегментом в payload → Telegram + CRM-таблица. Трекинг: `b2b_segment_select`, `b2b_lead`.

### 4.2. Вышивка (через ~2 мес) — подготовка архитектуры
- В модели Product/PrintType заложить `decoration_type` (dtf|embroidery|none) вместо хардкода DTF; в конструкторе — переключатель техники с разной ценовой матрицей (вышивка дороже, зоны меньше); в Telegram-уведомлениях — поле техники. Сделать сейчас в рамках Custom Print 2.0 — дешевле, чем переделывать потом.

### 4.3. DTF как отдельный источник прибыли (Блок 7)
- Пока без поддомена: раздел `/dtf/` («друк DTF на замовлення — для брендів і майстерень»): печать чужих макетов рулонами/листами. Калькулятор: размер × тираж → цена/м. Отдельная воронка лидов с пометкой dtf. SEO: «dtf друк україна», «dtf печать на заказ» — проверить спрос в GSC/Keyword Planner после запуска страницы.

### 4.4. Годовщина 1 июля (Блок 10) — план за 2 недели до
- **Техбаза:** (1) промокоды/скидки — проверить, есть ли механизм купонов в orders; если нет — простая модель Coupon (код, %, лимит, срок); (2) реферальная программа MVP: персональная ссылка `?ref=`, скидка обоим, учёт в заказе; (3) подтверждение заказа в Telegram-боте.
- **Маркетинг:** дроп «1 рік TwoComms», конкурс UGC в IG (фото в мерче, приз — худи), ретаргетинг на собранные аудитории (пиксельная база уже копится), email/TG-рассылка по клиентам, сторис-механики.
- **Минимальный пакет если мало времени:** промокод TWOYEAR −15% + IG-конкурс + ретаргет на ViewContent/ATC аудитории.

### 4.5. Цветовая схема (из Блока 5)
- Не делать редизайн «по вкусу». Сначала: (1) Clarity-данные по героям/CTA после месяца чистого трекинга; (2) A/B акцентного цвета CTA (текущий vs gold #D4AF37 vs orange) через GTM-эксперимент или серверный split. Black-gold традиционно работает на «premium streetwear», но это гипотеза для теста, не вера.

## 7. Реальный CAC — без иллюзий

«Конверсия по $4», которую ты видел вначале — это **дешёвое событие** (переписка/просмотр), а не покупка: пиксель был сломан, оптимизация шла на дешёвые действия.

**Реальная картина:**
- Всего за период реклама дала **~30–50 продаж** (оценка владельца).
- $941 / 40 продаж = **$24 ≈ 1 060 грн** средний CAC. Диапазон **$19–31 (850–1 400 грн)**.
- Рабочая часть — **messaging-воронка**: $397 / ~25 закрытых ≈ **$16/продажа**. Это и был реальный двигатель.
- При $50/день шло 6–15 диалогов, 2–3 продажи/день → CAC ~$17–25.

**Сопоставление с экономикой (CAC ≈ 1 000 грн):**
- Худи @1750, вклад 603 − 1000 = **−397** на холодную одиночную продажу.
- → Даже худи на холодном трафике при таком CAC **в минус**. Прибыль появлялась только когда: (а) диалог закрывался в комплект/дороже, (б) клиент возвращался, (в) часть «бесплатной» органики приписывалась рекламе.

**Цель: довести CAC до $10–15 (450–680 грн)** — реально через: рабочий пиксель (готово) + ретаргетинг тёплых + видео + оформленный Instagram. Тогда худи (603) и комплект (684) выходят в плюс.

## 10. B2B и консигнация — где реальные деньги (и где ловушка)

### Консигнация (модель твоего магазина в Харькове) — самый перспективный канал
- Факт: ~100 единиц «под реализацию» → **~10 000 грн/мес** без рекламных затрат (CAC ≈ 0).
- Если это ~13 проданных/мес по ~770 грн, твой вклад ≈ 13 × (770 − 569) ≈ **2 600 грн/мес с магазина**.
- **Масштабирование: 5 таких магазинов ≈ 50 000 грн выручки/мес, ~13 000 грн вклада/мес, без рекламы.** Это самый капитал-эффективный путь к объёму (нужен оборотный капитал на печать + риск зависших остатков).
- **Действие:** systematize — стандартный договор консигнации, ассортимент-хит (по 100 ед.), помесячная сверка, поиск 4–6 партнёров в других городах (Львов, Киев, Днепр, Одесса).

### Прямой опт + менеджер — только при крупных заказах
- План был: менеджер 15 000 база + 5%, KPI 2 конверсии/нед (~8 заказов/мес).
- При заказах по 8 ед. и опте ~650 грн: вклад в батче ≈ 650 − 520 − 32(5%) ≈ **98 грн/шт** → заказ 8 шт = ~784 грн → 8 заказов = ~6 300 грн/мес. **Не покрывает даже базу 15 000.**
- **Вывод (ты сам это увидел):** менеджер окупается только при средних заказах **30–50+ единиц** и/или дозаказах. **KPI привязать к ОБЪЁМУ (единиц/выручке), а не к числу заказов.** Нанимать — когда воронка B2B даёт ≥40–50 ед/нед потенциала.
- Оптовый «пол» цены: футболка **≥600–650**, худи **≥1350–1450** (не «лишь бы продать»).
- ГО/фонды −25%: следить, чтобы не уходить ниже COGS (худи −25% от 1750 = 1312 ≈ себестоимость 1147 — почти в ноль; это благотворительность из маржи, считать осознанно).

## 14. Как дать мне доступ в Meta Events Manager (ответ на вопрос #4)
