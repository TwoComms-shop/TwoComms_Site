## CRO-001. Соответствие концепции «difficulties» (АУДИТ ВЫПОЛНЕН, 05.07.2026)

Метод: чтение `twocomms_django_theme/templates/pages/index.html` (881 строка) + живой рендер https://twocomms.shop/ через headless-браузер (hero, категории, первые карточки).

### Что В ПОРЯДКЕ

- **Спорт/бейсбол-лексики НЕТ** ни в шаблоне главной, ни в живом рендере (grep по «спорт|бейсбол|baseball|sport» — 0 совпадений).
- Hero-акцент «з кодом продовження» + микролинк «Що означають дві коми?» — работают на бренд-концепцию (две запятые = продолжение, не точка).
- Бейджи «Засновано ветераном», «Народжений у Харкові» — соответствуют tone of voice, «я»-лексики на главной нет.
- Категорийный ряд начинается с «Лонгсліви» (порядок: Лонгсліви → Футболки → Худі → Кастом).

### НАХОДКИ (несоответствия позиционированию)

1. **Футболки везде на первом месте в копирайте.** Живой `<title>`: «TwoComms — стріт та мілітарі одяг з Харкова: **футболки, худі**»; hero-subtitle: «**Футболки**, худі та кастомний одяг…» (index.html:167); заголовок витрины: «Новинки каталогу — **футболки**, худі, лонгсліви» (index.html:571). Позиционирование «лонгсливы и худи первичны» нарушается на уровне слов. Фикс — переупорядочить перечисления: «Лонгсліви, худі та футболки…» (3 строки шаблона + title в SEO-блоке).
2. **Первые карточки витрины — футболки.** Живой рендер: первые 5 позиций — `twocomms-beliveidea-ts`, `ts-...-reality-bends-mentol`, `...dark-neon-edition`, `...future-2026`, `kha-edition-ts`; первый худи (`hd-...`) только 6-й. Порядок задаётся `apply_public_product_order` = `order_by('-priority','-id')` (catalog_helpers.py:130–137), т.е. управляется drag-and-drop админки, НЕ кодом. Фикс — владельцу/исполнителю поднять priority лонгсливов/худи в админ-панели (без кода). Связь с CRO-013.
3. **Концепция «труднощі/подолання» на главной не выражена текстом.** Hero транслирует «продовження», но слова-якоря концепции (труднощі, подолання, «кома, а не крапка») отсутствуют в видимом копирайте. Это НЕ баг, а зазор с концептуальным брифом — решение о усилении копирайта за владельцем (задача копирайтеру, не разработчику).
4. Категорийный ряд: «Футболки» стоят ВТОРЫМИ (перед Худі). Если порядок категорий тоже управляется приоритетом — поменять местами Худі и Футболки в админке.

Вердикт: критичных нарушений нет; 3 контент-правки (2 в шаблоне, 1 в админке) — исполнителю/владельцу.

## CRO-002. Работа всех CTA-кнопок главной (АУДИТ ВЫПОЛНЕН, 05.07.2026) — НАЙДЕН КРИТИЧЕСКИЙ БАГ

Метод: headless-браузер (desktop 1366×900 и mobile 360×740), hit-test каждой CTA через `elementFromPoint`, curl-проверка всех целевых URL.

### Что В ПОРЯДКЕ

- **Все целевые URL живые:** `/catalog/`, `/custom-print/`, `/pro-brand/`, `/catalog/long-sleeve|tshirts|hoodie/`, цветовые чипы `/catalog/?color=...`, карточки товаров — все отвечают **200 с 0 редиректов**.
- **Desktop 1366px:** hero-btn-primary, hero-btn-secondary, hero-microlink — все кликабельны (hit-test пройден), навигация работает.
- JS-ошибок в консоли при загрузке главной не зафиксировано.

### КРИТИЧЕСКИЙ БАГ (P0): на мобильном ≤360px hero-CTA полностью обрезаны и некликабельны

**Симптом (воспроизведён на живом сайте, viewport 360×740):**
- `.hero-section` имеет computed `height: 444px` (= 60vh), `overflow: hidden`, `contain: layout size`, при этом `.hero-content` фактически **727px** высотой.
- Нижние ~283px hero-контента — кнопки «Дивитись каталог», «Створити свій принт», микролинк «Що означають дві коми?» и все три бейджа — **физически обрезаны** `overflow:hidden` и не видны пользователю вообще (скриншот: после subtitle сразу идёт survey-блок).
- Hit-test: точка клика кнопок перекрыта `section#survey-section` (survey top=562px против кнопки 556–606px) — click попадает в survey, а не в CTA. Клик по `.hero-btn-primary` невозможен.
- Дополнительно точку клика на первом экране перекрывает PWA-prompt «Встановити TwoComms як застосунок?» (`.web-push-prompt`, `position:fixed`, `z-index:1060`, занимает 336×327px = ~45% экрана 360px).

**Корень (найден в коде):** инлайн critical-CSS в `base.html` (~строки 517–531):
```css
.hero-section { min-height: 60vh; height: 60vh; max-height: 60vh; contain: layout size; overflow: hidden; }
```
Комментарий в коде говорит, что это CLS-фикс (Phase 18, 2026-05-10). Фиксированная высота под десктоп сломала мобильный hero: на 360px контент выше 60vh.

**Задача исполнителю (P0, ~5 строк CSS):** в `base.html` добавить мобильный override:
```css
@media (max-width: 767px) {
  .hero-section { height: auto; max-height: none; contain: layout; }
}
```
(`contain: size` при height:auto недопустим — схлопнет секцию; оставить только `layout`.) После фикса перепроверить CLS mobile (CRO-003) — страховать высоту через `min-height`, а не `max-height`.

**Вторичная задача (P1):** PWA-prompt показывается сразу на первом экране и закрывает пол-экрана — отложить показ (например, после 2-й страницы или 30s) и не показывать поверх hero. Влияние на отказы paid-трафика прямое.

**Гипотеза, объясняющая цифры БД:** ATC=44 при 36k views и CTR 0,12% — мобильные пользователи (основной трафик из IG/TikTok!) не видят ни одной CTA на первом экране; это кандидат №1 в причины провала воронки наряду с бот-шумом.

## CRO-004. Отложенный GTM vs атрибуция paid-трафика (АУДИТ ВЫПОЛНЕН, 05.07.2026)

Метод: чтение боевого кода `base.html` (строки ~927–1051) и `static/js/analytics-loader.js`.

### Фактическая архитектура загрузки трекинга

1. **GTM (GTM-PRLLBF9H)** грузится по первому interaction (click/pointerdown/keydown — на главной БЕЗ scroll!) ИЛИ по таймауту: **главная 25–35s**, прочие роуты 4–9s, passive-режим 12–18s (Phase 22e, антилайтхаус-настройка).
2. **analytics-loader.js (~55KB — Meta Pixel, TikTok, Clarity)** — тоже только по первому interaction, fallback **25s**.
3. Ранний стаб `window.trackEvent` буферизует события до загрузки loader-а (fix 2026-06-12) — внутренний server-side трекинг НЕ теряется.

### Подтверждённые проблемы

1. **UTM/fbclid fast-path ОТСУТСТВУЕТ:** grep по `utm_|fbclid|gclid|ttclid` в инициализационном коде base.html — 0 совпадений. Пользователь с рекламы, который ничего не нажал и ушёл (классический bounce из IG/TikTok in-app браузера) — **невидим для Pixel/GA4/TikTok полностью**: ни PageView, ни ViewContent. На главной даже scroll не триггерит загрузку (interactionEvents для home = click/pointerdown/keydown).
2. Следствие для рекламных кампаний: Meta/TikTok недосчитываются верхней воронки → алгоритмы оптимизации получают меньше сигналов → дороже CPM/CPA. Оценить долю потерянных сессий можно после бот-фильтра (AN-раздел) сравнением SiteSession против GA4 sessions.
3. fbclid→_fbc cookie создаётся только внутри analytics-loader.js (строка ~111) — т.е. тоже только после interaction; если пользователь кликнул рекламу, посмотрел и ушёл — _fbc не создан, CAPI-события позже уйдут без него (связка с AN-013).

### Задача исполнителю (P1, ~10 строк JS в base.html)

В оба деферред-лоадера (GTM и analytics-loader) добавить fast-path: если `location.search` содержит `utm_|fbclid|gclid|ttclid` ИЛИ referrer — instagram/tiktok/facebook/google ads, грузить трекинг сразу (или с малым таймаутом 2s), игнорируя interaction-гейт. PageSpeed-замеры Лайтхауса это не заденет (LH не заходит с utm-метками) — антилайтхаус-логика Phase 22e сохраняется для organic.

### Acceptance

Открыть `https://twocomms.shop/?utm_source=audit&fbclid=test123` в чистом профиле, НЕ трогать страницу 5s → в Network должны появиться gtm.js и analytics-loader.js; cookie `_fbc` создана. Сейчас — не грузятся до interaction/25s.
