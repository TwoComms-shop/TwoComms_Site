## CRO-010. Фильтрация лонгсливов/худи

**Где проверено:** `storefront/views/catalog.py` (1215 строк), `storefront/services/color_filter.py`, `pages/catalog.html`, живые URL.

**Факты (подтверждено живой БД):**
- Категории: `long-sleeve` id=5, `hoodie` id=4, `tshirts` id=3 — совпадает с чек-листом. Все `is_active=True`, все `order=0`.
- Живые URL все 200 без редиректов: `/catalog/long-sleeve/`, `/catalog/hoodie/`, `/catalog/tshirts/`, `/catalog/?color=black`, `/catalog/theme/military/`. Несуществующий slug (`/catalog/futbolki/`) → корректный 404.
- Товаров published: hoodie 24, tshirts 23, long-sleeve 18 (65 published + 3 archived = 68).

**Архитектура фильтров:**
1. **Категория** — отдельные URL `/catalog/<slug>/` (view `catalog(cat_slug)`), фильтр-рейка `catalog-filter-rail` — обычные `<a>`, полная перезагрузка.
2. **Цвет** — `?color=slug1,slug2` (до 10 slug, санитизация `[a-z0-9-]`), OR-логика, chips из `build_available_colors()`. Состояние живёт в URL → ссылка шарима. Toggle-URL чипа корректно сохраняет прочие GET-параметры и сбрасывает `page` (`_build_chip_url` исключает `color` и `page`). Если для (категория, цвет) есть published `CategoryColorLanding` — чип ведёт на SEO-лендинг `/catalog/<cat>/<color>/` вместо `?color=`.
3. **Размер — ФИЛЬТРА НЕ СУЩЕСТВУЕТ.** Ни в view, ни в шаблоне, ни в моделях нет фильтрации по размеру. Чек-лист предполагал «фильтры по … размеру» — их нет физически. Размеры вообще не являются сущностью каталога (см. CRO-014).

**Отклонение от ожидания чек-листа:** «без полной перезагрузки» — НЕ выполняется: и категория, и цвет — полные page-load (server-rendered `<a>`). AJAX-фильтрации нет. При page-cache 600s для анонимов перезагрузка дешёвая (TTFB ~0.9–1.9s live), но UX-переходы медленнее SPA-паттерна.

**Вывод для исполнителя:** фильтры работоспособны и URL-шаримы; задачи — (а) решить, нужен ли размер-фильтр (требует введения складского учёта по размерам, см. CRO-014), (б) опционально AJAX-обновление грида.

## CRO-011. Пагинация каталога — 2 подтверждённых бага

**Где проверено:** `storefront/pagination.py`, `catalog.py`, `pages/catalog.html` (строки 421–457), живой HTML.

### БАГ №1 (P1): пагинация сбрасывает фильтр цвета
Шаблонная пагинация каталога рендерит ссылки как голый `href="?page={{ N }}"` (catalog.html:426,440,446). Query string заменяется целиком → `?color=` теряется.

**Живое подтверждение:** на `https://twocomms.shop/catalog/?color=black` (16 товаров, есть page 2) ссылки пагинации = `?page=2`, `?page=3` — БЕЗ `color=black`. Клик по «стр. 2» выкидывает пользователя из фильтра на нефильтрованный каталог. То же коснётся любых будущих GET-фасетов.

**Фикс-подсказка:** генерировать ссылки с сохранением GET (например `{% querystring page=N %}` Django 5.1+, или собранный в view base_qs), по аналогии с `_build_chip_url`, который это уже делает правильно для чипов.

### БАГ №2 (P1, SEO): `/catalog/?page=2..N` — индексируемые дубли без товаров
На корне каталога (`show_category_cards=True`) секция товаров НЕ рендерится вовсе (catalog.html:373 `{% if not show_category_cards %}`), но paginator по всем товарам всё равно создаётся в view.

**Живое подтверждение:** `/catalog/?page=2` → 200, title «Каталог TwoComms — … — сторінка 2», canonical `https://twocomms.shop/catalog/?page=2`, robots `index,follow`, rel prev/next присутствуют, а в DOM — **3 showcase-карточки категорий и 0 товаров**, т.е. контент идентичен `/catalog/` (page=1). SEO-комментарии в шаблоне (2026-05-16) утверждают «каждая страница пагинации показывает уникальный срез товаров» — для корня каталога это неверно: товарного грида там нет. Google получает 5 индексируемых near-duplicate страниц + rel=next с корня на пустышку.

**Фикс-подсказка:** на корне каталога либо не строить paginator/prev-next вовсе (нет грида — нет пагинации), либо 301 `?page>1` → `/catalog/` (по аналогии с уже существующим 301 `/?page=1` → `/` в home()).

### Что работает корректно
- Категорийные страницы: `/catalog/hoodie/` 16 товаров + `?page=2` 8 товаров, **пересечение id между страницами = 0** (порядок `-priority,-id` детерминирован, дублей нет).
- rel prev/next и self-canonical на категорийной пагинации корректны (`hoodie/?page=2`: canonical на себя, prev на чистый `/catalog/hoodie/`).
- Legacy-путь `/catalog/page/N/` → redirect на `?page=N` (urls.py:84).
- Out-of-range `?page=99` → 200 c контентом последней страницы и canonical на фактический номер (paginator.get_page клампит) — допустимо.
- `?page=1` на корне НЕ редиректится на чистый URL (в отличие от home) — микродубль, canonical при этом чистый; P3.

## CRO-012. Lazy-load изображений каталога

**Где проверено:** `partials/product_card.html`, `storefront/templatetags/responsive_images.py` (`optimized_image`, default `loading="lazy"`), живой HTML `/catalog/hoodie/`.

**Замер живого HTML (16 карточек):**
- width="1080" height="1600" — на **всех 16** карточках → CLS-безопасно. Подтверждает CRO-003 (CLS 0.00).
- `loading`: 2 eager + 14 lazy. Шаблон жёстко задаёт eager только `forloop.counter0 < 2` (catalog.html:388). Чек-лист рекомендует 4–8 eager для первого экрана: на desktop-гриде 4 колонки первый ряд = 4 карточки, из них 2 грузятся lazy → потенциальная задержка LCP-соседей. P3-рекомендация: eager для первых 4.
- `<picture>` у всех 16, но **AVIF-sources только у 7 из 16**; у 9 карточек `src=/media/product_colors/N.webp` без srcset вовсе — для файлов из `media/product_colors/` responsive-варианты (`_320w/_480w/_640w…`) не сгенерированы (у `media/products/optimized/` — сгенерированы). Эти 9 карточек грузят полноразмерный webp на 45vw-мобиле. Пересекается с CRO-022; задача исполнителю: прогнать генератор вариантов по `product_colors/`.

## CRO-014. Статусы наличия в каталоге — механики наличия не существует

**Где проверено:** `storefront/models.py` (Product), `productcolors/models.py` (ProductColorVariant), `partials/product_card.html`, живая БД.

**Факты:**
- У Product НЕТ полей склада. Единственный рубильник — `status` (`published`/`archived`/…), управляется вручную; каталог фильтрует `status='published'` везде (+ archived скрыты полностью, phantom-SKU в гриде невозможны).
- У ProductColorVariant ЕСТЬ поле `stock` (PositiveInteger, default=0), НО живая БД: **75 из 75 вариантов stock=0**. Поле мертво: никто его не заполняет, каталог/карточка/чекаут его не читают (grep по views: 0 использований stock в логике выдачи).
- Размерного склада нет вообще (размер — не сущность, а строка в OrderItem).
- Карточка каталога не имеет ни бейджа «нет в наличии», ни состояния out-of-stock — при 100% stock=0 честная разметка наличия сейчас невозможна.

**Риск:** товар, реально закончившийся, остаётся кликабельным и заказываемым, пока владелец вручную не снимет с публикации → CRO-047 (заказ на отсутствующий товар) не может быть предотвращён системно.

**Задача исполнителю (связка с TECH-010):** решение по инвентаризации: (а) вести stock на уровне вариант×размер (новая модель) ЛИБО (б) официально зафиксировать модель «под заказ/print-on-demand — всё published всегда доступно» и убрать мёртвое поле stock. До решения — бейджи наличия в каталоге не внедрять (будут врать).

## CRO-015. N+1 при рендере каталога — НЕ ПОДТВЕРЖДЁН (норма)

**Где проверено:** боевой сервер, `manage.py shell`, Django test client + `CaptureQueriesContext` (cache-buster `?_audit=`, обход page-cache; DEBUG не включался — см. бонус-находку).

| `/catalog/hoodie/` | 19 |

## Бонус-находки (вне пунктов, зафиксировать)

1. **`?sort=` — мёртвая ветка.** `catalog.html:47` ставит noindex при `request.GET.sort`, SEO-комментарии описывают `?sort=new|sale|price_asc|price_desc`, но ни один view не читает `sort` (grep по storefront/ = 0). Сортировки для пользователя не существует; шаблонная ветка — рудимент. Либо реализовать сортировку, либо убрать упоминания.
2. **DEBUG=True роняет боевой рендер.** При включении DEBUG в shell-сессии рендер любой страницы падает с `compressor.exceptions.OfflineGenerationError` (offline-манифест собран под DEBUG=False). Следствие: диагностика с DEBUG на бою невозможна в принципе; любые инструкции «включи DEBUG на проде» — вредны. Для замеров использовать `CaptureQueriesContext`.
3. **Поиск (`/search/`) рендерит ВСЕ результаты без пагинации** (`product_list = list(filtered_search_qs)`, paginator не создаётся). При 65 товарах не больно, но паттерн не масштабируется; также `search()` обёрнут в широкий `except Exception` с фолбэком-пустышкой (глотает ошибки — вписывается в CB-020).
4. **Fragment-cache ключи** грида включают `request.get_full_path` → страницы с/без фильтра кэшируются раздельно, утечки между состояниями нет (проверено по ключу `catalog_products_home_cards_v5_webp_swap`).
