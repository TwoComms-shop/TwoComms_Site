## АРХИТЕКТУРА КОРЗИНЫ (реверс-инжиниринг) — контекст для всех пунктов ниже

Корзина хранится в **сессии** (`django.contrib.sessions`, backend `cached_db`, cookie 30 дней):
- `request.session['cart']` — обычные товары. Ключ позиции: `"{product_id}:{size}:{color_variant_id|'default'}[:{fit_option_code}]"`. Значение: `{product_id, size, color_variant_id, fit_option_code, fit_option_label, qty}`.
- `request.session['custom_print_cart']` (константа `SESSION_CUSTOM_CART_KEY`) — кастом-принт позиции (по lead_id).
- `request.session['promo_code_id']` — применённый промокод.

**Важно:** цена НИКОГДА не хранится в сессии — везде берётся из `Product.final_price` (актуализация цен, защита от манипуляций). Это верное решение (подтверждено в `view_cart`, `add_to_cart`, `update_cart`, `cart_summary`, `cart_mini`, `calculate_cart_total`).

**Синхронизация между устройствами (для залогиненных):**
- Модель `accounts/cart_models.py::UserCart` (OneToOne с user) — снапшот `cart_data` + `custom_cart_data` + `promo_code_id`.
- `accounts/cart_middleware.py::CartSyncMiddleware` (26-й в цепочке MIDDLEWARE, `settings.py:221`):
  - `process_request` → `hydrate_session_from_db` (тянет БД→сессию, если ревизия БД отличается);
  - `process_response` → `persist_session_to_db` (пишет сессию→БД только при статусе 2xx/3xx и реальном изменении).
- `accounts/cart_sync.py` — ядро синхронизации. Ревизия строится **по содержимому (sha256), а не по `updated_at`** — это правильно (грубое разрешение datetime в некоторых СУБД). Снапшот делается через `copy.deepcopy` (views мутируют `cart[key]['qty']` in-place — без deepcopy сравнение «было/стало» сломалось бы). Это грамотная деталь.
- `accounts/cart_signals.py` — на сигнал `user_logged_in` вызывает `merge_session_into_db` (мердж гостевой корзины с БД). Сигнал зарегистрирован в `accounts/apps.py::ready()` (подтверждено).

Все функции синхронизации оборачивают исключения в `try/except` + `logger.debug/warning`, чтобы синхронизация никогда не валила запрос. Оборотная сторона — ошибки синхронизации глотаются молча (согласуется с CB-020: 697 широких except).

## CRO-030. Логика мини-корзины vs full-page корзины (АУДИТ ВЫПОЛНЕН) — архитектура ОК, декомпозиция нужна

| View | URL | Что отдаёт | Кэш |
|---|---|---|---|
| `view_cart` (`cart.py:433`, алиас `cart`) | `/cart/` | Полная HTML-страница корзины + чекаут-форма | `@never_cache` (подтверждено live: `Cache-Control: no-cache, no-store, private`, `Vary: Cookie`) |
| `cart_mini` (`cart.py:1347`) | `/cart/mini/` | HTML-partial выпадающей мини-корзины (`partials/mini_cart.html`) | без явного кэша, `Vary: Accept-Language, Cookie` |

Обычные товары: `_merge_standard_carts` — при совпадении ключа `product_id:size:color:fit` **количества суммируются** (не дублируются). Кастом: `_merge_custom_carts` — union по lead_id (без удвоения). Промокод: приоритет у только что применённого в гостевой сессии, иначе из БД. **Логика верная.** (Живой тест логина не проводился — нужен тестовый аккаунт; логика подтверждена статически и покрыта юнит-тестами по данным CB-024.)

### НАХОДКА CRO-030-A (P2, тех. долг): `cart.py` — 1850 строк, кандидат на декомпозицию
Файл смешивает: рендер корзины, add/update/remove, промокоды, custom-cart, cart_summary/mini, contact_manager, Nova Poshta city/warehouse search. Согласуется с CB-022 (god-files). Не баг, но повышает риск регрессий при правках воронки (деньги).

### НАХОДКА CRO-030-B (P2): дублирование логики «очистки мёртвых товаров»
`cart_summary` (`cart.py:1290+`) и `cart_mini` (`cart.py:1354+`) содержат почти идентичный блок удаления из корзины товаров, которых уже нет в БД (`missing_products`). `view_cart` же просто `continue` пропускает такие позиции, НЕ вычищая их из сессии. → Несогласованность: на странице `/cart/` мёртвый товар остаётся в сессии (просто не показывается), а `/cart/summary/` его вычистит. Рекомендация исполнителю: единый хелпер очистки.

## CRO-031. Баги добавления/удаления (АУДИТ ВЫПОЛНЕН) — **P0 + P1 ПОДТВЕРЖДЕНЫ ЖИВЫМИ ТЕСТАМИ**

### 🔴 P0-БАГ (CRO-031-A): «Отравление корзины» → перманентный HTTP 500 на `/cart/` и `/cart/mini/`

**Суть:** `add_to_cart` (`cart.py:776`) берёт `color_variant_id` из POST **без валидации** и кладёт сырое значение в сессию:
```
color_variant_id = request.POST.get('color_variant_id')   # cart.py:783 — НЕТ int()/проверки
...
key = f"{product.id}:{size}:{color_variant_id or 'default'}"
item = {... 'color_variant_id': color_variant_id, ...}
request.session['cart'] = cart                              # в сессию попадает 'abc'
```
Сам `add_to_cart` при этом отвечает `200 OK` (внутри `int(color_variant_id)` обёрнут в try/except → `color_variant_int=None`, offer_id строится).

НО последующий рендер корзины падает, т.к. `int()` вызывается **без защиты**:
- `view_cart` (`cart.py:572`): `color_variant = color_variants_map.get(int(color_variant_id)) if color_variant_id else None` — цикл ловит только `except Product.DoesNotExist` (`cart.py:619`), поэтому `ValueError: invalid literal for int() with base 10: 'abc'` **пробрасывается → 500**.
- `cart_mini` (`cart.py:~1354+`): `variant_ids = [i.get('color_variant_id') ...]` → `ProductColorVariant.objects.in_bulk(['abc'])` и `int(variant_id)` → тоже падение.

**Живое подтверждение (боевой сервер, 05.07.2026, анонимная сессия):**
```
POST /cart/add/  product_id=1&size=S&qty=1&color_variant_id=abc   → HTTP 200 {"ok": true, "count": 5, ...}
GET  /cart/                                                        → HTTP 500 (Server Error)
GET  /cart/mini/                                                   → HTTP 500 (Server Error)
GET  /cart/summary/                                                → HTTP 200 {"ok": true, "count": 5, "total": 3940.0}  (не падает — не делает int())
```
**Последствия:**
1. Пользователь с «отравленной» сессией **не может открыть свою корзину** и **не может оформить заказ** — 500 держится 30 дней (срок cookie), пока не очистит куки.
2. Восстановление возможно **только** через `POST /cart/clean/` (`clean_cart`), которого пользователь не видит (это AJAX-эндпоинт, `@require_POST`; `GET` даёт 405). Живой тест: `POST /cart/clean/` → 302 → после этого `/cart/` снова 200. Т.е. это чинится, но не самим пользователем.
3. Вектор доступен без авторизации и без CSRF-препятствий для самого пользователя (это его сессия) — фактически self-DoS/ловушка; также любой сторонний скрипт/битый фронт-стейт с некорректным `data-variant` вызовет это же.

**Первопричина:** отсутствие приведения/валидации `color_variant_id` к int (или к None) на входе `add_to_cart` **и** незащищённые `int()` при рендере. Достаточно нормализовать на входе (`try: int(...) except: None`) — тогда ключ станет `...:default`, и все рендеры переживут.
**Задача исполнителю:** санитизация `color_variant_id` (и `fit_option`) в `add_to_cart`; плюс defensive `int()` в `view_cart`/`cart_mini` (обернуть в try или расширить `except (Product.DoesNotExist, ValueError, TypeError)`); плюс миграция-хилер для уже отравленных сессий (при чтении — молча дропать позиции с нечисловым color_variant_id).

> **Примечание для владельца:** во время теста боевая анонимная сессия была отравлена и затем **очищена** тем же `POST /cart/clean/` — сейчас тестовая корзина пуста, реальные данные не затронуты (тест шёл в изолированной анонимной cookie-сессии, без логина).

### 🟠 P1-БАГ (CRO-031-B): быстрый двойной клик / гонка → задвоение количества

**Суть:** кнопка «Додати в кошик» (`main.js:1669` — делегированный `click` по `[data-add-to-cart]`) **не блокируется на время in-flight запроса** и не дебаунсится. Каждый клик = отдельный `fetch('/cart/add/')`. `add_to_cart` делает `item['qty'] += qty` — два быстрых клика по одному варианту дают qty=2.

**Живое подтверждение (05.07.2026):**
```
add #1 (M):                     count=1
add #2 (M, «двойной клик»):     count=2      ← та же позиция, qty удвоилось
parallel add ×2 (L и XL):       count=3, count=4   ← гонка обработана (session cached_db), но без защиты от намеренного спама
```
Это ровно тот сценарий из чек-листа («быстрый двойной клик не должен создавать 2 позиции»). Для одного варианта это «+1 к qty», что пользователь может не заметить и купить лишнее.
**Также:** т.к. пиксельное событие шлётся в `.then()` после каждого успешного ответа (`trackAddToCartAnalytics`, `main.js:1750`), двойной клик = **2 события AddToCart** в GA4/Meta/TikTok (искажение CTR view→cart, релевантно факту add_to_cart=44 из аудита БД).
**Задача исполнителю:** дизейблить кнопку/ставить in-flight-флаг на время запроса (idempotency), либо дебаунс; на бэке — опционально явно принимать «set qty», а не только «+=».

### 🟡 P2 (CRO-031-C): нет верхней границы количества
`update_cart` (`cart.py:887`) проверяет только `qty < 1` (→400). Верхней границы нет — `qty=999999999` принимается, `line_total = price*qty` раздувается, а также нет проверки склада (см. CRO-014/CRO-025: stock везде 0, механики наличия нет). `add_to_cart` тоже без потолка (`qty = max(qty, 1)`). Риск: абсурдные суммы в корзине/чекауте, потенциальная нагрузка. **Задача:** разумный потолок (напр. ≤ 99 на позицию) + дружелюбное сообщение.

### Удаление — корректно
`remove_from_cart` (`cart.py:951`) поддерживает удаление по exact key и по `product_id+size`. Быстрое двойное удаление безопасно (`cart.pop(k, None)` идемпотентен). Удаление последней позиции обнуляет счётчик (проверяется через `cart_summary`/`get_cart_count` после операции). Отдельных багов удаления не выявлено.

## CRO-032. Кэширование не ломает мини-корзину (АУДИТ ВЫПОЛНЕН) — ОК

**Где проверено:** `twocomms/cache_headers.py`, `media_cache_middleware.py`, `storefront/cache_signals.py`; живые HTTP-заголовки.

**Живой замер заголовков (05.07.2026):**
```
GET /cart/       → Cache-Control: max-age=0, no-cache, no-store, must-revalidate, private ; Vary: Cookie, Accept-Language, Accept-Encoding
GET /cart/summary/ → Vary: Accept-Language, Cookie (без публичного кэша)
GET /            → Vary: Cookie, Accept-Language, Accept-Encoding (страница кэшируема, но динамика подтягивается AJAX-ом)
```
**Вывод:** страницы с корзиной (`view_cart`) помечены `@never_cache` (`cart.py:432`) + `private` — из общего page-cache другому пользователю НЕ отдаются. `cart_summary`/`cart_mini` варьируются по `Cookie`. Счётчик корзины подтягивается AJAX-ом **после** загрузки (кэшируемой) страницы: `main.js` откладывает `refreshCartSummary()`/`refreshMiniCart()` до пользовательского взаимодействия или таймаута (2000–6500ms в зависимости от route/device — `main.js:159+`). SSR-маркер `window.__TC_SYNC_CART` не даёт дёргать `/cart/summary/`, если серверу известно, что корзина пуста. **Архитектура корректна, багов кэша мини-корзины не выявлено.**

**Побочно (info):** отложенный до 2–6.5s AJAX-рефреш бейджа означает, что вернувшийся пользователь с непустой корзиной видит бейдж не мгновенно (компромисс ради PageSpeed/TBT, согласуется с CRO-004). Не баг воронки, но UX-нюанс.

## CRO-033. Событие add_to_cart — сервер + пиксели (АУДИТ ВЫПОЛНЕН) — двоение из-за CRO-031-B

**Где проверено:** `cart.py:840` (`record_add_to_cart`), `main.js:356` (`pushAddToCartEvent`), `main.js:1535/1641` (`trackAddToCartAnalytics` → `window.trackEvent('AddToCart', payload)` + dataLayer).

**Норма «одно добавление = одно серверное UserAction + одно событие в каждый пиксель»:**
- Серверная часть: `add_to_cart` вызывает `record_add_to_cart` ровно один раз на запрос (обёрнуто в try/except с warning). ОК на один запрос.
- Клиентская часть: `trackAddToCartAnalytics(d, btn, qty)` шлётся один раз в `.then()` успешного ответа. ОК на один запрос.
- **НО из-за CRO-031-B (нет блокировки кнопки) двойной клик = 2 запроса = 2 серверных UserAction + 2 пикс. события AddToCart.** Это прямой вклад в искажение метрик воронки.
**Живой тест Meta/TikTok Test Events не проводился** (нужны доступы к Events Manager — у владельца). Рекомендация исполнителю: после фикса CRO-031-B прогнать Meta Test Events / TikTok Test Events и сверить, что 1 клик = 1 событие Server (CAPI) + 1 Browser (Pixel), с общим `event_id` для дедупа CAPI↔Pixel (связано с §0 TECH-06x про единое определение событий).

## CRO-034. Кастом-принт позиции в корзине (АУДИТ ВЫПОЛНЕН) — split реализован, UI-объяснение требует проверки

**Где проверено:** `storefront/custom_print_config.py` (`SESSION_CUSTOM_CART_KEY`), `checkout.py` (split approved/pending, строки 41–248), `_collect_custom_cart_state` (`cart.py:350`).

**Логика подтверждена (код):**
- `checkout.py` разделяет кастом-позиции на **approved** (присоединяются к заказу — `approved_custom_leads`) и **pending** (остаются в сессии — `pending_custom_keys`). Строки 46–64.
- Если платить нечем (нет обычных товаров И нет approved-кастома) — «wait» (строка 65–66): pending-кастом НЕ блокирует, но и заказ не создаётся, пока нечего оплачивать.
- После оформления: обычная корзина чистится, approved-кастом прикреплён к заказу, **в сессии остаются только неапрувнутые кастом-позиции** (`checkout.py:239–248`) — пользователь сможет оплатить их позже. **Корректно.**
- В `view_cart` (`cart.py:710–729`): при наличии кастома оплата разрешена **только если ВСЕ кастом-позиции approved** (`payment_allowed = all_approved`), передоплата (`prepay_200`) при кастоме запрещена (`prepay_allowed = False`, менеджер решает финальную цену). Флаги `any_pending_review/any_awaiting_review/any_rejected/all_approved/has_draft_items` прокидываются в шаблон.

**НАХОДКА CRO-034-A (P2, требует UI-проверки):** логика гейтинга оплаты есть, но **понятность объяснения пользователю в `cart.html`** (почему кастом «ждёт», что делать) не верифицирована в этом заходе — нужен визуальный аудит `pages/cart.html` (2577 строк) с реальной pending-кастом позицией. Связано с фактом БД: `CustomPrintLead` = 28, **все в статусе `new`** (воронка кастома мертва, разрыв №3) — т.е. реальные пользователи застревают на этапе `new`, и объяснение статуса в корзине критично. **Задача исполнителю:** проверить копирайт/бейджи статуса кастома в корзине вживую и сверить с реальными статусами лидов.

## CRO-035. Восстановление корзины (АУДИТ ВЫПОЛНЕН) — для залогиненных ОК, для гостей ограничено cookie

**Где проверено:** `accounts/cart_middleware.py`, `cart_sync.py`, отчёт `CART_RESTORATION_REPORT.md` (корень репо), `settings.py` (SESSION).

- **Залогиненные:** корзина переживает закрытие браузера и смену устройства — хранится в `UserCart` (БД), гидратируется на каждом запросе (`hydrate_session_from_db`). **Подтверждено кодом.** Механизм ревизии по sha256 корректно ловит изменения с другого устройства.
- **Гости:** «восстановление» = только срок жизни session-cookie (`SESSION_COOKIE_AGE = 30 дней`, `settings.py:977`; engine `cached_db`). Очистка cookie/приватный режим = потеря гостевой корзины. Это ожидаемо, но именно поэтому CRO-031-A (отравление) особенно болезненно для гостей — единственный «сброс» одновременно теряет всё.

**НАХОДКА CRO-035-A (P2, гигиена репозитория):** в корне репозитория лежит `CART_RESTORATION_REPORT.md` (и ещё ≥5 cart-отчётов: `CART_FIXES_REPORT.md`, `CART_DISPLAY_BUG_FIX.md`, `CART_DISPLAY_FIX_REPORT.md`, `CART_CONSULTATION_MODAL_FIX.md`, `MINI_CART_MONO_CHECKOUT_FULL_PAYMENT_FIX.md`). Согласуется с CB-002 (202 .md в корне — история фиксов вместо docs/archive). Эти отчёты числят фиксы «сделанными» — их фактическую актуальность (особенно относительно найденного P0 CRO-031-A) исполнителю следует перепроверять, а не доверять на слово.
