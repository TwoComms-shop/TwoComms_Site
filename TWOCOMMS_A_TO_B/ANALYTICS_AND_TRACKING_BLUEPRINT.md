## 1. Что система должна позволять понимать

Про каждого посетителя/покупателя (в рамках законного и разумного):
- откуда пришёл (канал, кампания, ad set, объявление, креатив);
- на какую страницу пришёл; какие страницы/товары смотрел; сколько времени реально был активен; куда скроллил; что нажимал;
- какие размеры, цвета, товары, фильтры смотрел;
- где начал и где бросил оформление;
- сколько раз вернулся;
- написал ли в Direct; оформил ли заказ; оплатил ли; выкупил ли; вернулся ли за повтором;
- какая реклама дала не клики, а **CM2 на доставленных заказах**.

## 2. Честные границы (обязательный раздел)

| Категория | Примеры | Политика |
|---|---|---|
| Можно и законно | first-party события, серверные события, UTM/click ID, CRM-статусы, агрегаты | собирать |
| Требует согласия | cookies, пиксели, heatmaps/session replay | cookie consent (до согласия — только строго необходимое), privacy policy обновить |
| Нельзя / не стоит | скрытый fingerprinting, тексты личных сообщений, пароли, платёжные данные, PII в параметрах событий | запрещено |
| Только приблизительно | time-on-page, кросс-девайс, view-through, скролл как «внимание» | помечать как оценку |
| Не обещать | «точность до секунды», 100% атрибуция Direct | не строить решения на ложной точности |

Heatmaps/session replay (если внедрять): маскирование всех полей ввода и PII, исключение страниц оплаты/чекаута с данными, включение только при согласии.

Time-on-page: считать по heartbeat активности (видимая вкладка + событие взаимодействия в интервале), а не по разнице timestamps открытия страниц. Открытая фоновая вкладка ≠ внимание.

## 3. Архитектура (целевая)

```text
Браузер:  GA4 (через GTM) · Meta Pixel · TikTok Pixel (при запуске)
          dataLayer (единая schema) · consent-модуль
Сервер:   Meta Conversions API + дедупликация (event_id)  [уже есть — аудит]
          TikTok Events API (при запуске)
          UTM/click-ID capture: utm_*, fbclid→fbc, fbp, ttclid,
          gclid — сохраняются в серверную сессию и в БД   [частично есть — аудит]
БД:       session_id ↔ visitor_id ↔ customer_id ↔ order_id ↔ CRM-статусы
          (единая цепочка идентификаторов)
CRM-слой: статусы заказов (§5 MASTER_PLAN_V2), статусы лидов,
          теги источника диалогов, sale_source словарь
Дашборд:  ежедневный/еженедельный/ежемесячный (см. §8)
Governance: UTM-конвенция, event schema, словарь sale_source, PII-политика
```

### 3.1 Идентификаторы и связка
| Идентификатор | Где рождается | Где хранится | С чем связывается |
|---|---|---|---|
| session_id | первый hit сессии | серверная сессия + БД | pageviews, события |
| visitor_id | first-party cookie (с согласием) | БД | сессии между визитами |
| utm_* / fbclid / fbc / fbp / ttclid / gclid | параметры/cookie клика | БД, привязаны к сессии | атрибуция канала |
| customer_id | регистрация/заказ | БД | все заказы человека |
| order_id | создание заказа | БД | статусы, суммы, CM |
| conversation_ref | ref-параметр ссылки в Direct / ad_id из webhook / ручной CRM-тег | CRM | реклама → диалог |

**Критический фикс (из аудита старого плана, не сделан):** при создании заказа копировать utm_*/click-ID из серверной сессии в Order; проставлять is_converted в UTM-сессии; двигать статусы CustomPrintLead. Это первые три пункта аудита.

## 4. Атрибуция Direct-продаж (почему UTM недостаточно)

Рекламный клик «в переписку» не проходит через сайт → UTM не возникает. Три слоя решения:

1. **Технический:** ref-параметры в ссылках на Direct (где возможно), ad_id/campaign_id из webhook входящего диалога (если платформа отдаёт), автоответ с кнопками — выбор кнопки уже сегментирует интент.
2. **Процессный:** CRM-тег источника проставляется в момент ПЕРВОГО ответа оператора (10 секунд дисциплины). Словарь sale_source фиксированный: `ads_meta / ads_tiktok / aio / ig_organic / tiktok_organic / custom / friends / marketplace / consignment / b2b_outreach / other`.
3. **Опросный:** self-reported source в диалоге/чекауте («звідки дізналися?») — хранится отдельным полем, не перезаписывает технические слои.

Модель атрибуции: хранить **first-touch** и **last non-direct touch** раздельно + self-reported. В отчётах показывать оба, не смешивать.

## 6. CRM-статусы и B2B-воронка

Заказы: `created → deposit_paid → paid → shipped → delivered / refused_rts / cancelled; return; defect_replacement` (единый словарь с §5 MASTER_PLAN_V2). Каждая смена статуса — timestamp.

Кастом-лиды: `new → in_progress → quoted → won / lost(причина)`.

B2B: `lead_found → first_touch → replied → qualified → file_received → quoted → sample → paid → repeat / lost(причина)` + поля: сегмент, канал, потенциал (м/мес или ед/мес), метры/единицы, вклад, дата follow-up.

## 7. UTM governance

- Шаблон для всех платных ссылок: `utm_source=fb|tiktok&utm_medium=paid&utm_campaign={{campaign.name}}&utm_content={{ad.name}}` (динамические макросы платформ).
- Organic-посты — свои utm по конвенции; AI-источники (chatgpt.com, perplexity.ai, gemini, claude.ai) детектить по referrer как отдельные источники.
- Referrer-спам (binance, wp-login, xmlrpc) — в is_bot/исключения (известный баг — в аудит).
- Запрещено: свободный формат utm «от руки»; таблица-реестр всех активных utm обязательна.
